AI 工程改造四级纲领

蓝色光标汽车事业部 | AI工程队作战指南 v2.0 | 2026.05.17

架构总览

AI工程改造按业务颗粒度分四级，自上而下：业务级（整条业务线的AI产品化）→ 客户级（一个完整客户的全面AI改造）→ 项目级（特定事件类型的标准化Skill）→ 工作流级（单项生产能力的AI化）。

四级不是割裂的——工作流级是原子单元，项目级是工作流的编排组合，客户级是对一个客户所有项目/工作流的全面扫描和改造，业务级是跨客户的系统产品。底层支撑所有级别的是共享基础设施层。

原则：不是所有工作流都要AI化。全面扫描后精准判定——能自动化的自动化，不能自动化的明确AI介入比例和人工补充比例。

Level 1 业务级系统：达芬奇Auto
定位

围绕达人营销投放业务，构建跨客户的系统级AI产品。不是提效工具，是客户锁定系统——用累积的私有投放数据和客户知识形成切换成本。

类比：贝壳把每套房子的真实交易数据结构化→反向定义行业标准。达芬奇Auto把每个达人的真实投放效果结构化→反向定义达人营销选号标准。

三层能力（优先级递降）
  1. 能力1：客户知识引擎 — 每个客户的历史知识、反馈、项目经验训练成知识库，输出80分以上策略方案。把人脑中的客户认知结构化，不随人员流动丢失。
  2. 能力2：投放数据驱动匹配 — 基于星合全量达人数据+两年私有投放数据（CPM/CPE/A3/达人配合度），升级标签系统，提高选号适配度。这是达芬奇"数据世界B"——星合20人没接上的那层。
  3. 能力3：全链路自动化 — 选号到结算全流程自动化。优先级让位于1+2，南迁人力+RPA可覆盖大部分。

1+2 = 命门（定价权+商业模式）。3 = 锦上添花。

纲领要求
  • 逐客户构建：零跑先行 → 理想/小鹏复制 → 3客户以上跨客户洞察层浮现 → 平台化
  • 数据资产意识：每个客户的改造过程本身就是在存数据资产，不是慢是必须
  • 技术路径：真正的AI工程（RAG/embedding/知识库/prompt工程/效果评估闭环），非RPA

Level 1与其他级别的关系

Level 1面向达人投放业务（BU收入的40%+），是独立的产品线。Level 2-4面向公关/社会化/创意业务（BU收入的60%），形态完全不同。两棵树共享一个根（基础设施层），但产品目标和商业模式各自独立。

Level 2 客户级系统：全面AI改造
定位

围绕一个完整的客户服务生态系统，全面扫描所有工作流，逐个判定AI化方案：

  • 能全自动化的 → 直接自动化（数据采集/格式转换/日报生成）
  • 能大幅提效的 → AI出初稿+人精修（脚本/规划/报告）
  • 需要人工主导的 → 明确AI辅助比例（策略/谈判/创意概念）
  • 纯人工的 → 不碰（客户关系维护/现场执行/紧急公关）

不是要求所有环节AI化，而是全面摸清后精准适配。

执行标准
  1. 全面摸排 — 对客户所有工作流做一次完整诊断（已有方法论：AI改造业务摸排工作手册）
  2. 量化工时 — 统计各工作流占团队总工时的比例，画出人力消耗地图
  3. 分类定级 — 每个工作流标注：全自动/AI主导/AI辅助/纯人工
  4. 优先级排序 — 先做"工时占比高+AI化可行性高"的（落在右上象限的）
  5. 客户场景诊断 — 判断属于信息成熟型/信息混沌型/中间型，决定投入深度
  6. 迭代闭环 — 部署后持续采集反馈，按周迭代Skill，建立飞轮

纲领要求
  • 进入客户级改造前，必须完成完整的业务摸排（四步流程/三个必问/诊断报告）
  • 产出物：客户AI改造地图（全工作流 × AI化程度矩阵） + 改造路线图（按优先级排期）
  • 不追求100%AI化——有些环节人做就是更好/更安全/更有温度，明确标注"不改造"
  • 每个客户建独立知识库（品牌档案/人物小传/历史作品/客户偏好/反馈日志）
  • 客户级改造完成标志：团队人均产能提升50%+ 或 项目人力减半不影响交付质量
客户级的核心价值

客户级改造的产出不只是效率提升——过程中积累的客户知识库、客户偏好模型、自动化工作流本身就是Level 1（达芬奇Auto）的数据资产。每改造一个客户，Level 1的知识引擎就厚一层。两个级别互相喂养。

Level 3 项目级系统：事件类型标准化
定位

围绕特定的事件类型/项目类型，建立可跨客户复用的标准化AI工作流。比如"发布会"作为一种事件类型，不管是理想还是奕派的发布会，底层工作流结构是一致的（只是填充的知识库不同）。

项目级改造的产出是：一套事件类型Skill包——接入任何一个客户的知识库，就能产出该客户该事件类型下的全套内容。

Skill包结构（以"新车发布会"为例）
  1. 输入采集Skill — 从活动方案/产品文档/议程表中提取结构化信息（时间/地点/参会人/核心卖点/禁忌）
  2. 新闻稿生成Skill — 基于结构化信息+品牌知识库+历史高分样本 → 生成通稿/分稿
  3. 讲话稿生成Skill — 基于领导风格模型+本次核心信息 → 生成讲话稿
  4. 社媒内容日历Skill — 基于活动时间线+各平台调性 → 生成多平台内容排期
  5. 媒体Q&A生成Skill — 基于可能提问+官方口径+禁区 → 生成应答手册
  6. 质检Skill — 交叉比对全套物料一致性（数据/称谓/时间/信息不矛盾）
纲领要求
  • 每个事件类型拆解为标准工作流清单（不遗漏、不冗余）
  • 每个工作流对应一个独立Skill，可单独调用也可串联为Agent链
  • Skill与知识库解耦：同一套Skill接入理想知识库=理想发布会内容，接入奕派知识库=奕派发布会内容
  • 测试标准：每个Skill包配套3-5个历史Case作为回归测试基准
  • 版本管理：Skill包按"事件类型_v版本号"管理，每次迭代有变更日志
  • 产出评估：首版产出质量不低于70分（可直接用作修改基础），迭代后不低于80分

Level 3与Level 2的嵌套关系

Level 3产出的Skill包是Level 2的积木。客户级改造 = 诊断该客户涉及哪些事件类型 → 调用对应Skill包 → 接入该客户知识库 → 补充客户特有的定制化需求。Level 3做得越标准，Level 2推广到新客户时越快。

定位

最小颗粒度的AI改造单元——单项内容生产能力的AI化。是Level 2/3的原子组成部分，同时也可独立运作（比如"热点监测"不依赖任何特定客户或项目）。

  • 明确标注：该工作流适用场景+不适用场景+人工兜底条件
  • 周级迭代：投入使用后按飞轮机制每周更新（有新反馈时）

Infra 共享基础设施层
定位

支撑所有四个级别的底层能力。不重复造轮子——每个级别的改造项目共享同一套基础设施，在此之上构建各自的应用逻辑。

知识库架构规范
  • 每客户一个独立知识库目录（品牌档案/人物小传/历史作品/客户偏好/模板库）
  • 客户知识库双份存储：本地完整版（含敏感判断）+ 星合品牌空间净化版（供达芬奇调用）
  • 行业知识库共享维护（行业趋势/平台动态/竞品数据/通用最佳实践）
  • 知识库新增与Skill规则更新分离：事实信息直接追加，写作规则需AI提炼+去重+测试
  • 格式统一：markdown文件为主，结构化数据用JSON/YAML，长文本用独立文档
Skill开发规范
  • 统一命名：[客户]_[事件类型]_[工作流]_v[版本号]
  • 统一结构：角色设定 → 任务描述 → 知识库引用 → 输出格式 → 禁忌清单 → 质检要求
  • 测试标准：每个Skill至少3个历史Case，关键词覆盖率+禁用词检测+格式检查全通过
  • 迭代节奏：有反馈时每周更新一次，无反馈时保持稳定不动
  • 版本管理：每次更新保留上一版可回滚，变更日志记录新增/修改/删除的规则

四级嵌套关系

递归关系

Level 1 业务级 包含→ Level 2 客户级 包含→ Level 3 项目级 包含→ Level 4 工作流级
  • Level 4（工作流）是最小单元，可独立运作，也嵌入任何上层
  • Level 3（项目）= 一组工作流的有序编排（如发布会 = 新闻稿+讲话+社媒+Q&A）
  • Level 2（客户）= 该客户所有项目/工作流的全面改造矩阵
  • Level 1（业务）= 跨客户的系统产品，吃Level 2积累的数据资产

数据流向

  • 向上喂养：Level 4产出→Level 3验证→Level 2积累知识→Level 1形成跨客户数据资产
  • 向下赋能：Level 1跨客户洞察→Level 2改造方案→Level 3标准Skill包→Level 4执行
  • 基础设施层同时服务所有级别，不分先后

启动优先级

  • 立即启动：Level 4（多个工作流已在进行）+ Level 2零跑（摸排完成启动改造）
  • 本月启动：Level 3日常公关Skill包标准化 + Level 2千里/现代摸排
  • W22+启动：Level 1 API对接 + Level 3发布会Skill包（等下一场活动）
  • 基础设施层持续建设，不设独立启动节点
