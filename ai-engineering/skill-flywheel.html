Skill 自动迭代飞轮

从"一次性训练"到"持续学习" | 技术方案 v1.0 | 2026.05.17
飞轮逻辑
一线日常使用AI出稿 → 客户反馈/修改意见 → 自动提炼规则
↓
注入Skill新版本 → 自动测试验证 → 下周出稿质量提升
↓
越用越准 → 客户越离不开
滴滴验证过的模式

滴滴稿件创作中心已实现：每周一凌晨3点 → 取最近100条审改记录 → DeepSeek提炼规则 → 写入WritingRule表 → 下次生成时自动注入prompt。同样的逻辑，需要适配到"无服务器、Cherry Studio前端"的轻量场景。

两套实现方案
方案A：轻量版（飞书 + 定时脚本）
  • 反馈收集：飞书表格
  • 知识库：飞书文件夹
  • 训练脚本：本地Python定时任务
  • 适用：信息成熟型客户（理想/现代）
  • 成本：零，现有工具即可
  • 部署时间：1天
方案B：标准版（服务器 + 自动化）
  • 反馈收集：Web表单或飞书机器人
  • 知识库：服务器文件系统
  • 训练脚本：服务器Cron自动执行
  • 适用：信息混沌型客户或多客户统一管理
  • 成本：服务器 ¥56/月（可复用滴滴同一台）
  • 部署时间：3-5天
选择建议

先用方案A启动（零成本验证飞轮是否能转），跑通后再决定是否升级到方案B。 方案A的核心约束是需要一线手动填飞书表格记录反馈——如果他们不填，飞轮就断了。方案B把反馈收集嵌入工作流（审稿时自动留存），不依赖人的自觉性。

方案A 详细设计：飞书 + 定时脚本
架构总览

三个飞书组件 + 一个定时Python脚本，组成完整闭环：

组件用途更新频率
飞书表格「反馈记录」一线每次收到客户修改意见后填入实时
飞书文件夹「知识库/新增」新确认稿件、新产品信息、新规范文档投入有新材料就扔
飞书文档「Skill版本」存放当前Skill的完整文本 + 版本日志每周自动更新
Python训练脚本每周定时拉取新数据 → AI提炼 → 更新Skill每周一凌晨
Step 1：反馈收集表设计

飞书表格，列结构如下：

字段类型说明
日期日期自动填充
稿件类型单选新闻稿/讲话稿/社媒/周报...
客户原始意见文本客户的修改批注/反馈原文（截图或文字）
修改了什么文本一线实际改动的要点概述
修改原因分类多选事实错误/风格不符/结构问题/信息缺失/客户偏好/其他
是否为新规则单选是/否（标记：这个修改代表一条新的"客户规则"吗？）
填写人人员自动

关键：让填表零负担。每次客户有修改意见就花2分钟记一条。不要求写长文，关键词+原文截图即可。

Step 2：每周训练脚本逻辑

Python脚本，每周一凌晨3点执行（macOS launchd 或服务器cron）：

# skill_weekly_train.py — 每周自动迭代Skill # 执行时间：每周一 03:00 import json from datetime import datetime, timedelta from lark_sdk import LarkClient # 飞书SDK from openai import OpenAI # DeepSeek兼容 # ============ 1. 拉取本周新数据 ============ def fetch_weekly_feedback(sheet_token, since_date): """从飞书表格拉取上周的新反馈记录""" client = LarkClient() rows = client.sheets.read(sheet_token, range="反馈记录!A:G") new_rows = [r for r in rows if r['日期'] >= since_date] return new_rows def fetch_new_knowledge_files(folder_token): """从飞书文件夹拉取新增的知识库文件""" client = LarkClient() files = client.drive.list_files(folder_token, order_by="modified_time") # 只取最近7天新增/修改的 cutoff = datetime.now() - timedelta(days=7) return [f for f in files if f['modified_time'] > cutoff] # ============ 2. AI提炼规则 ============ def extract_rules(feedback_records): """用AI从反馈记录中提炼可复用规则""" client = OpenAI(base_url="https://api.deepseek.com", api_key="...") prompt = f"""你是一个品牌文案规则提炼专家。 以下是本周客户的修改反馈记录（共{len(feedback_records)}条）。 请从中提炼出可复用的写作规则，格式为： - 规则描述（一句话） - 触发场景（什么情况下适用） - 正确做法 - 错误示例（从反馈中提取） 只提炼确实代表"客户标准"的规则，忽略一次性的偶发修改。 反馈记录： {json.dumps(feedback_records, ensure_ascii=False, indent=2)}""" response = client.chat.completions.create( model="deepseek-chat", messages=[{"role": "user", "content": prompt}], temperature=0.3 ) return response.choices[0].message.content # ============ 3. 更新Skill ============ def update_skill(current_skill, new_rules, new_knowledge_summary): """将新规则和知识融入现有Skill""" client = OpenAI(base_url="https://api.deepseek.com", api_key="...") prompt = f"""你是Skill维护工程师。请将以下新规则和新知识整合到现有Skill中。 要求： 1. 新规则插入到"客户偏好规则"章节 2. 去重：如果新规则与现有规则含义相同，合并而非重复添加 3. 如有冲突（新规则与旧规则矛盾），以新规则为准（代表客户最新标准） 4. 新知识（新产品/新话术）更新到知识库引用说明中 5. 保持Skill整体结构不变，只做增量更新 当前Skill： --- {current_skill} --- 本周新规则： {new_rules} 本周新增知识摘要： {new_knowledge_summary}""" response = client.chat.completions.create( model="deepseek-chat", messages=[{"role": "user", "content": prompt}], temperature=0.2 ) return response.choices[0].message.content # ============ 4. 测试验证 ============ def run_test_cases(updated_skill, test_cases): """用历史测试用例验证更新后的Skill是否退化""" client = OpenAI(base_url="https://api.deepseek.com", api_key="...") results = [] for case in test_cases: response = client.chat.completions.create( model="deepseek-chat", messages=[ {"role": "system", "content": updated_skill}, {"role": "user", "content": case['brief']} ] ) output = response.choices[0].message.content # 简单检查：关键词覆盖率、禁用词检测 results.append({ 'case': case['name'], 'passed': check_quality(output, case['expected_keywords'], case['banned_words']) }) return results # ============ 5. 主流程 ============ def main(): since = (datetime.now() - timedelta(days=7)).strftime('%Y-%m-%d') # 拉取数据 feedback = fetch_weekly_feedback(SHEET_TOKEN, since) new_files = fetch_new_knowledge_files(FOLDER_TOKEN) if not feedback and not new_files: notify("本周无新反馈和新知识，Skill保持不变。") return # 提炼规则 new_rules = extract_rules(feedback) if feedback else "无新规则" new_knowledge = summarize_files(new_files) if new_files else "无新知识" # 读取当前Skill current_skill = read_skill_from_feishu(SKILL_DOC_TOKEN) # 更新Skill updated_skill = update_skill(current_skill, new_rules, new_knowledge) # 测试验证 test_results = run_test_cases(updated_skill, TEST_CASES) all_passed = all(r['passed'] for r in test_results) if all_passed: # 写回飞书 + 记录版本 write_skill_to_feishu(SKILL_DOC_TOKEN, updated_skill, version_note=f"自动更新 {datetime.now().strftime('%Y-%m-%d')}, 新增{len(feedback)}条反馈规则") notify(f"Skill已自动更新 ✓ 本周提炼{count_new_rules(new_rules)}条新规则，测试全部通过。") else: # 测试未通过，标记需要人工review notify(f"⚠️ Skill更新后测试未通过，需人工确认。失败用例：{[r['case'] for r in test_results if not r['passed']]}") if __name__ == "__main__": main()

方案B 详细设计：服务器 + 自动化采集
与方案A的核心差异

方案A依赖一线主动填表记录反馈。方案B把反馈收集嵌入工作流——一线用AI出稿、改稿的全过程自动留存，不需要额外动作。

架构
组件技术说明
写作前端Web App（类似滴滴方案）一线在这里写稿/改稿，全部操作自动记录
数据库SQLite / PostgreSQL存储：稿件版本、修改diff、审核意见、AI输出
训练Cron服务器定时任务每周一凌晨自动提炼规则，更新Skill
Skill版本管理Git仓库或数据库表每次更新记录版本，可回滚
测试引擎Python脚本更新后自动跑测试用例，不通过则回滚
自动采集的数据来源（无需一线主动填表）
  1. AI初稿 vs 人工终稿的diff — 系统自动对比，提取"人改了什么"
  2. 审核意见文本 — 审核人（组长/客户）写的批注直接入库
  3. 修改次数和位置 — 统计哪些段落被改最多，暗示Skill在哪里不准
  4. 客户一版通过率 — 作为Skill质量的北极星指标
方案B的杀手优势

方案A飞轮能否转取决于一线是否愿意填表（现实是：忙起来一定不填）。方案B的数据采集是工作流的副产品——一线正常写稿改稿，系统自动留存一切。这是滴滴方案证明过的：飞轮能持续转，不是因为人自觉，而是因为数据采集嵌入了无法跳过的流程节点。

方案B的触发条件

以下任一出现，就该升级到方案B：

  • 一线持续不填反馈表（飞轮断了）
  • 管理3个以上客户的Skill，手动维护成本过高
  • 需要把系统给客户使用（于总个人操作系统等场景）
  • 需要精确的"一版通过率"等量化指标
关键设计决策
Q: 自动更新会不会把Skill改坏？

三层保护：

  1. 测试用例卡口 — 更新后跑3-5个历史Case，不通过则回滚
  2. 版本管理 — 每次更新保留上一版，随时可回退
  3. 人工审核窗口 — 前期先设为"生成草案→人工确认→正式发布"，跑稳后再改为全自动
Q: 一周一次够不够？频率该多高？

一周一次是对的。 理由：

  • 客户偏好不会天天变，周级别积累反馈再统一更新，避免过拟合到个别意见
  • 每次更新需要测试验证，频率太高维护成本高
  • 团队需要时间适应"上周的Skill和这周的不一样了"
  • 例外：品牌规范大改（如换logo、换slogan）时手动触发一次即时更新
Q: 知识库更新和Skill规则更新有什么区别？
知识库更新Skill规则更新
内容事实性信息（新产品/新数据/新话术）写作规则（客户偏好/风格/结构）
来源新文件入库客户修改反馈
更新方式直接追加（不需AI处理）需AI提炼+去重+整合
风险低（加信息不影响原有规则）中（新规则可能与旧规则冲突）
验证人工检查一次即可必须跑测试用例
Q: 和达芬奇Auto的关系？

文字类Skill飞轮 = 达芬奇Auto"能力1（客户知识引擎）"的子集。两者共享底层逻辑：

  • 客户知识随使用积累 → 系统越来越懂客户 → 客户切换成本递增
  • 文字类先跑通验证飞轮机制，验证成功后复制到达人选号/投后分析等场景
  • 最终统一为：每个客户一个"客户知识引擎"，文字只是其中一个输出通道
