文字类工作 AI 工程改造

全流程工作指南 | AI工程队内部标准文档 | v1.0 2026.05.17
全流程七阶段
1 立项诊断 → 2 素材收集 → 3 Skill训练 → 4 人工校准 → 5 测试达标 → 6 前端部署 → 7 运维迭代
核心认知

绝大多数文字类工作都适合AI改造。真正影响成败的不是"AI能不能写"，而是"喂给AI的信息是否齐全、准确、结构化"。写作从来不是瓶颈，信息供给才是。

1
立项诊断
1-2天
目标

判断该客户/场景属于哪种类型，决定投入深度和产品形态。

三类场景判断（决策树）
场景类型客户特征投入深度实例
信息成熟型 客户信息供给及时完整、对口人专业稳定、服务时间长有默契 轻量：Skill + 知识库即可 理想汽车
信息混沌型 客户内部信息不通/延迟、对口人反复焦虑、Brief不清 重：Skill + 知识库 + 前端产品（信息拉齐系统） 东风奕派
中间型 流程规范但灵活度有限、偶发临时状况、投产比不值得上系统 中等：Skill + 知识库，人工兜底偶发 现代中国
诊断问题清单（30分钟访谈即可）
  1. 这个客户的文案类需求有哪几种？（新闻稿/讲话稿/社媒/周报...）
  2. 每种稿件的月产量和平均耗时？
  3. 客户给的Brief通常完整吗？信息到位率大概多少？
  4. 一篇稿件从Brief到终稿，通常改几版？卡在哪一步？
  5. 团队中谁在用AI？用到什么程度？
  6. 客户的品牌规范/禁忌事项多不多？有成文的文档吗？
经验教训

不要在诊断阶段就排除任何客户。"文字资料少"不是不能做AI的理由（现代中国证明了流程规范比资料量更重要），"Brief不清"也不是（奕派证明了这是需要前端产品解决的问题，不是放弃AI的理由）。

阶段交付物

一份一页纸的《项目立项卡》：客户名 / 场景类型 / 文案种类 / 月产量 / 一线负责人 / AI种子用户 / 预判投入深度

2
素材收集
3-5天
目标

收齐训练Skill所需的全部原始材料。这是整个流程中最容易被低估的环节——素材质量直接决定Skill质量。

标准素材收集清单（每个客户必收）
  • 品牌规范文档 — 品牌手册、VI规范、文案规范、禁用词列表、话术标准
  • 客户偏好记录 — 对口人/决策者的审稿习惯、常改的点、喜恶倾向（从一线口述整理）
  • 过往确认版稿件 — 至少10-20篇最终确认版（不是过程稿），按稿件类型分类
  • 客户官方信息库 — 产品参数、标准话术、领导层公开讲话、官网/官媒内容合集
  • 过往Brief样本 — 客户下发的Brief原始文档，越多越好，含最终确认版的对照
  • 审核反馈记录 — 客户批注/修改记录/退稿原因汇总（能体现客户标准的核心材料）
  • 行业/竞品参考 — 客户认可的竞品稿件、行业标杆案例
  • 一线经验总结 — 资深文案的口述经验（"这个客户喜欢什么/忌讳什么/踩过什么坑"）
收集方法
  • 指定一线对接人（最好是服务最久的文案），给他清单，约定3天内交齐
  • 不要让一线自己整理——给飞书文件夹，按清单建好子目录，让他们直接扔进去
  • 品牌规范和客户偏好如果没有成文文档，安排30分钟语音访谈，AI工程队负责整理成文
  • 过往稿件优先收集"一版过"或"只改一次就过"的高质量样本
关键发现（来自实战）

理想：服务6年+，资料极其丰富，孙鹏超本人就是行走的知识库。收集几乎零阻力。
现代中国：外企流程规范，每次活动都有标准Brief模板和审核流程文档，资料自带结构化。
奕派：资料少且碎，Brief经常不清楚，但这不意味着不能做——意味着需要在Phase 6补一个"信息拉齐"前端。Skill训练可以先用有的材料训，后面边用边补。

阶段质量门

满足以下任意一条即可进入下一阶段：(1) 确认版稿件达到10篇以上 + 品牌规范齐备；(2) 稿件不足10篇但有详细的一线经验口述记录 + 客户反馈记录。不要等到"材料完美"——先训练，训练过程中暴露的缺失再补。

3
Skill训练
2-3天
目标

基于收集的素材，训练出能产出60-70分稿件的初版Skill。注意：这一步产出的不是完美Skill，而是"能用来收集反馈的初版"。

Skill结构（标准模板）
  1. 角色定义 — 你是[客户名]的[稿件类型]写手，服务该客户[X]年
  2. 品牌规范注入 — 直接嵌入品牌规范/禁用词/标准话术（作为系统指令的一部分）
  3. 风格样本 — 嵌入3-5篇代表性确认版稿件作为few-shot示例
  4. 客户偏好规则 — 从审核反馈中提炼的"客户在意什么/不在意什么"规则列表
  5. 输入输出格式 — 定义用户输入的Brief格式，和期望的输出结构
  6. 知识库引用 — 指向外挂知识库（产品信息/历史稿件/行业数据）的调用方式
训练要点
  • 一个客户可以有多个Skill（按稿件类型拆分：新闻稿Skill、讲话稿Skill、社媒Skill...）
  • 知识库和Skill分离——知识库存事实（产品信息/数据），Skill管写法（风格/结构/规则）
  • 初版不求完美，先跑出能看的东西，然后靠Phase 4的人工校准迭代
  • 每个Skill附带2-3个"测试用例"：用真实的历史Brief作为输入，跑一遍看输出
实例：理想汽车新闻稿Skill
知识库：理想汽车全系产品参数 + 李想/马东辉/刘杰公开讲话全文 + 2025-2026全部确认版新闻稿
Skill规则：标题必须含核心产品点 + 第一段交代时间地点人物事件 + 数据必须准确引用知识库 + 禁止使用"赋能""生态""矩阵"
孙鹏超在此基础上已能独立使用，日常用Cherry Studio调用
阶段交付物

初版Skill文件 + 知识库文件 + 3个测试用例的输出样本。打包发给一线业务组。

4
人工校准
3-5天
目标

让一线业务团队看AI产出的稿件，指出"哪里不对"，AI工程队据此迭代Skill。这一步是把60-70分提到80分的关键。

校准流程
  1. 把测试输出发给一线文案 + 组长，要求他们像改稿一样批注（红字标出不对的地方）
  2. 收集批注后分类：(a)事实错误 → 补知识库 (b)风格不对 → 改Skill规则 (c)结构不符 → 改模板 (d)缺信息 → 补素材清单
  3. 修改Skill后重跑同样的测试用例，对比前后差异
  4. 重复1-3次，直到一线反馈从"不能用"变成"改改能用"
常见校准点（来自实战经验）
  • "口气不对" → 补充更多风格样本，或明确写入语气指令（正式/轻松/技术范/用户共情）
  • "这个数据不是最新的" → 知识库需要更新机制，标注数据有效期
  • "客户从来不这么说" → 品牌禁用词列表需要扩充，收集更多客户实际用语
  • "结构是对的但深度不够" → 可能需要行业知识补充，或引入竞品参考
  • "写得太安全了没亮点" → 调整Skill中的创意空间指令，加入创意方向引导
注意

校准的目标是80分，不是100分。100分永远需要人最后润色——这是不可消灭的，也是人的价值所在。如果一线说"改改能用"或"在这个基础上调整比从零写快多了"，就是达标了。

阶段质量门

一线文案确认：AI初稿在自己动手改30分钟以内能达到可提交客户的标准。如果还需要大改重写，说明Skill未达标，继续迭代。

5
测试达标
1周
目标

在真实业务中试跑Skill，验证稳定输出80分。不是实验室测试，是在真实项目中用起来。

测试方法
  1. 选择下一个即将到来的真实项目（不是虚构场景）
  2. 一线文案先用AI出初稿，再按正常流程修改提交客户
  3. 记录：(a)AI初稿耗时 (b)人工修改耗时 (c)客户反馈 (d)最终改版次数
  4. 对比基线：同类型稿件之前从零写需要多长时间
  5. 连续3篇以上稳定达标 = 通过
80分标准定义
  • 事实准确（产品信息、数据、人名无误）
  • 品牌规范合规（禁用词、话术、格式全部正确）
  • 结构完整（标题、导语、正文、结尾各部分齐全）
  • 风格匹配（读起来像这个品牌/客户的稿件，不是通用AI腔）
  • 只需局部润色即可提交（不需要推翻重写或大幅结构调整）
阶段交付物

《测试报告》：3-5篇真实项目的测试记录，含before/after时间对比、客户反馈、稳定性评估。通过后进入部署阶段。

6
前端部署与协同设计
1-2周
目标

根据诊断结果，为该团队配置合适的日常使用界面和多人协作流程。

前端方案选型（按场景）
场景类型推荐前端理由
信息成熟型 Cherry Studio + Skill作为知识库 一线AI能力强，给工具就够用，不需要额外界面
中间型 Cherry Studio + 飞书文档协同 流程规范，加一层飞书审批/评论即可覆盖协同需求
信息混沌型 定制前端（Brief收集系统 + AI写作台 + 审稿流） 核心问题是信息不通，需要产品级解决方案
协同问题调研（必须做）

文案工作通常2-3人协作（文案+AE+组长），需要调研清楚：

  • 谁发起？谁写？谁审？谁提交客户？
  • 多人同时修改同一篇稿件的情况多吗？还是串行流转？
  • 审核意见怎么传递？飞书评论？微信？口头？
  • 客户反馈回来后，谁负责改？改完谁再审？
  • 有没有版本混乱的痛点？（改了半天发现改的是旧版）

基于以上调研结果，决定是否需要加版本管理、评论流、审稿通知等协同功能。

实例：理想汽车当前方案
孙鹏超日常使用 Cherry Studio，内置理想汽车Skill + 知识库。出稿后直接贴入飞书文档，组内评论修改，终稿提交客户。不需要额外前端。协同流程已经够顺。
实例：奕派远期方案（待开发）
Skill训练可以先用当前材料，但真正解决问题需要：(1) Brief结构化收集前端——让奕派产品部/GTM部直接填表，自动生成结构化Brief (2) 于总个人操作系统——口述→成稿→自改 (3) 解决"人传人"信息损耗。这属于产品级工程，不在标准流程范围内，单独立项。
阶段交付物

一线团队开始日常使用AI工具产出稿件。有明确的使用界面、协同流程和常见问题FAQ。

7
自动迭代飞轮
持续运转
目标

Skill不是训练一次就完事的静态产物，而是一个持续学习的活系统。建立"使用→反馈→提炼→更新→使用"的自动化闭环，让Skill越用越准，客户越来越离不开。

飞轮运转逻辑

一线日常使用AI出稿 → 客户反馈/修改意见 → 自动提炼规则 → 注入Skill新版本 → 自动测试验证 → 下周出稿质量提升 → 循环

越用越准 → 客户越离不开 → 这就是锁定

封装内容（进入飞轮前必须完成）
  • Skill最终版 + 版本号 + 更新日志
  • 知识库内容清单 + 更新机制说明
  • 3-5个标准测试用例（用于验证每次自动更新是否退化）
  • 使用说明文档（1页纸，给一线看的）
  • 反馈收集渠道已建好（飞书表格或Web系统）
飞轮实现：方案A（轻量版 — 飞书 + 定时脚本）

适用：信息成熟型/中间型客户，零额外成本，立即可做

  • 反馈收集：飞书表格，一线每次收到客户修改花2分钟填一条（日期/稿件类型/客户意见/修改内容/原因分类）
  • 知识库更新：飞书文件夹，新确认稿件/新产品信息/新规范直接扔进去
  • 每周自动训练：Python脚本每周一凌晨自动执行——
    1. 拉取本周新反馈记录
    2. AI提炼可复用规则（去重、判断是否代表客户标准）
    3. 将新规则整合进现有Skill（增量更新，不破坏原有结构）
    4. 跑测试用例验证更新后Skill未退化
    5. 通过则自动发布新版，失败则通知人工review
  • 通知：飞书群消息"[客户名] Skill v2.3 已自动更新，本周新增3条规则"
飞轮实现：方案B（标准版 — 服务器 + 自动采集）

适用：信息混沌型客户，或方案A飞轮断裂时升级

  • 核心差异：不依赖一线主动填表。反馈采集嵌入工作流——一线在Web前端写稿/改稿，修改过程自动留存
  • 自动采集数据源：
    1. AI初稿 vs 人工终稿的diff（系统自动对比"人改了什么"）
    2. 审核意见文本（审核人批注直接入库）
    3. 修改频率和位置（统计哪些段落被改最多 = Skill在哪里不准）
    4. 客户一版通过率（Skill质量的北极星指标）
  • 参考实现：滴滴稿件创作中心已验证此模式——每周一凌晨3点，取100条审改记录，DeepSeek提炼规则，写入数据库，下次生成时自动注入prompt
方案选择与升级时机
先用方案A升级到方案B的触发条件
零成本验证飞轮是否能转一线持续不填反馈表（飞轮断了）
适合1-2个信息成熟型客户同时管理3个以上客户的Skill
1天即可部署需要把系统给客户直接使用
手动维护成本低需要精确的"一版通过率"等量化指标
落地三步走
阶段时间做什么
1. 手动飞轮立即开始建飞书反馈表 → 思琪每周手动整理 → 手动更新Skill
2. 半自动飞轮W22-W23Python脚本自动提炼规则，人工审核后更新
3. 全自动飞轮W24+Cron定时执行，自动更新 + 自动测试 + 自动通知
关键保护机制
  • 测试用例卡口：每次自动更新后跑3-5个历史Case，不通过则回滚上一版
  • 版本管理：每次更新保留历史版本，随时可回退
  • 人工审核窗口：前期"生成草案→人工确认→发布"，稳定后改全自动
  • 频率控制：一周一次（客户偏好不会天天变，避免过拟合个别意见）
为什么飞轮是战略级设计

飞轮不只是运维手段，它是客户锁定的核心机制。Skill越用越懂客户 = 客户切换成本递增 = 定价权。这和达芬奇Auto"能力1（客户知识引擎）"是同一个逻辑：文字类先跑通验证飞轮机制，验证成功后复制到达人选号、投后分析等所有场景。

知识库会腐烂

产品更新了、人事变了、品牌策略调了——如果知识库不跟着更新，Skill输出会越来越离谱。飞轮的价值之一就是：通过每周自动采集新材料、新反馈，让知识库自然保鲜，而不是靠人记得去更新。

并行能力

AI工程队当前配置（陈思琦 + 5名实习生），可同时推进2-3个客户的改造项目。信息成熟型客户可以交给实习生独立执行（Phase 2-5），信息混沌型需要思琪+王总介入产品设计（Phase 6）。

附录：三个项目的经验沉淀
理想汽车 — 标杆案例
  • 成功因素：服务时间长、资料丰富、孙鹏超AI能力强、客户信息供给好
  • Skill状态：基本完成，日常使用中
  • 剩余工作：协同流程优化（多人协作场景调研）
  • 可复制经验：信息成熟型客户的标准流程从这里提炼
现代中国 — 规范驱动
  • 成功因素：外企流程规范、Brief模板标准化、资料虽不多但结构清晰
  • Skill状态：初步训练完成，质量较高
  • 经验：不一定需要海量资料，"结构化程度高"比"数量多"更重要
  • 启示：规范的客户可以快速复制，2周内完成全流程
东风奕派 — 信息供给问题
  • 核心问题：不是Skill不好，是客户Brief不清、信息碎片化、反复修改
