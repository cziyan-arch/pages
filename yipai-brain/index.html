编辑大脑：公关文案信息中台

从"等信息→催信息→写稿"到"信息自动归位→缺口可见→一键出稿"
产品设计文档 · 2026年5月

一、要解决的问题

现状

47页活动方案PDF丢过来

周杨花2小时翻完、手动提取信息

发现价格没给，微信问江贝

江贝说"还在等产品部"

等3天，信息陆续到齐

最后2天赶稿，改3版

目标

47页方案上传

系统5分钟自动提取+结构化

自动标注：价格/权益待确认

自动推送江贝：请确认这4项

信息补齐后自动触发出稿

一版过，顶多改细节

核心洞察：写作只占整条链路20%的时间。80%的时间花在"等信息、找信息、判断信息够不够"上。编辑大脑要压缩的是这80%。

二、完整链路：从事件到成稿

1
信息输入
素材进入系统的所有入口
活动方案PDF/PPT → 飞书文档上传或转发
产品参数表 → 飞书文档/Excel
零散要求 → 微信聊天截图/语音转文字 → 人工贴入飞书
公开信息 → 系统自动抓取（工信部公告、竞品动态、行业新闻）
自动 人工转录微信/口头
2
信息提取与结构化
AI读完所有输入材料，自动生成结构化信息卡
从47页方案中提取：活动基本信息 / 产品数据 / 领导讲稿框架 / 传播规划
与知识库交叉比对：product_facts / brand_bible / writing_patterns
自动补充已有信息（如007闪现版在知识库中已有的基础参数）
全自动 AI处理
3
缺口分析 + 主动催办
对比"出稿需要什么"和"现在有什么"，找出差值
硬性缺口（无法出稿）：价格、核心权益、上市时间 → 推送给江贝/对接人
软性缺口（可以先写框架）：媒体最终名单、直播细节 → 标注但不阻断
推送方式：飞书机器人消息，清单式，每条标注"需要谁提供"
自动推送 江贝确认/补充
4
传播角度研判
AI基于信息+竞品+行业趋势，给出传播角度建议
不是等江贝说"写个什么角度"，而是AI主动给出2-4个角度建议
每个角度附：核心论点 / 新闻点 / 适用稿件类型
排优先级：差异化强度 × 信息成熟度 × 传播力
AI研判 江贝选择/调整
5
Brief自动生成
所有信息汇聚为一份结构化brief，直接可喂给Skill
按INTAKE.md的标准字段自动填充
已确认项 ✅ / 待确认项 ⚠️ / 缺失项 ❌ 状态清晰可见
出稿条件判断：信息完备度达标 → 自动触发；不达标 → 先出框架稿
全自动
6
Skill出稿
brief完整后，调用现有Skill系统生成稿件
自动选择稿件类型和模板（产品上市稿 / 活动稿 / 政策稿...）
自动适配审稿人偏好（江贝：产品点前置标题；王帅：开头必须有力）
输出：正文 + 质检报告 + 备选标题
AI生成
7
人工终审 + 迭代
周杨/思琦快速review，江贝审稿
重点只看：信息准确性、调性是否到位、有无敏感表述
修改意见回流系统 → 更新规则库 → 下次更准
江贝反馈的偏好变化 → 自动更新client_profiles

三、关键变化：谁做什么变了

环节 现在谁做 以后谁做
翻完47页方案提取信息 周杨（2小时） AI（5分钟）
判断信息齐不齐、缺什么 周杨凭经验 AI自动比对出稿模板
催江贝补信息 周杨微信追问 系统自动推送清单
想传播角度 周杨+江贝讨论 AI给建议，江贝选
写brief 周杨手动整理 AI自动生成
写稿 周杨写（3-5小时） AI生成（2分钟）+ 人工review（30分钟）
改稿3-5版 信息更新导致反复 信息齐了才出稿，一版过

一句话

从"人驱动的线性流程"变成"AI驱动的信息就绪系统"
人只做两件事：确认信息、审稿件

四、技术实现

平台选择：飞书

  • 客户内部文件已在飞书，无需迁移
  • 飞书多维表格作为信息中台（每场活动一行，每个字段对应brief必填项）
  • 飞书机器人推送缺口提醒和角度建议
  • 已有lark-cli配置，API权限齐全
  • 微信端的碎片信息：人工截图/复制 → 贴入飞书，短期无法自动化（微信API限制）

核心模块

模块A：信息提取引擎
  • 输入：PDF/PPT/DOCX/飞书文档/图片
  • 处理：Claude API解析 → 按预设schema结构化
  • 输出：结构化信息卡（JSON），写入飞书多维表格
  • 与知识库交叉：自动补充product_facts中已有的车型数据
模块B：缺口分析 + 催办
  • 维护一份"出稿必要信息checklist"（按稿件类型不同）
  • 每次信息更新后自动比对：已有 vs 必要
  • 缺口 → 飞书机器人消息推送，@相关人
  • 信息完备度打分（0-100%），达标阈值后自动触发下一步
模块C：编辑研判
  • 输入：结构化信息 + 竞品数据 + 行业趋势
  • 处理：Claude API分析 → 生成传播角度建议
  • 输出：2-4个角度，附优先级排序和理由
  • 可选：自动监测公开信息源（工信部/汽车之家/懂车帝），有事主动推送
模块D：Brief生成 + Skill对接
  • 从多维表格中读取信息 → 按INTAKE.md格式生成brief
  • 信息完备 → 自动调用SKILL生成稿件
  • 信息不完备 → 生成框架稿（缺失项用占位符）
  • 输出：Word文档 + 质检报告，推送到飞书群

五、落地节奏

Phase 1 · 1周
手动版验证
不写代码。用Claude Code手动跑完整个流程：方案进来 → 人工喂给AI提取 → 生成缺口清单和角度建议 → 出brief → 出稿。

目标：验证"编辑大脑"的输出是否真的对江贝/周杨有用。
Phase 2 · 2-3周
飞书Bot MVP
搭建飞书机器人：方案文件丢进飞书群 → bot自动提取 → 输出信息卡+缺口清单 → 推送到群里。

技术：lark-cli + Claude API + 飞书消息卡片。思琦可以做。
Phase 3 · 按需
完整产品
飞书多维表格信息中台 + 自动监测公开信息 + 缺口催办 + 一键出稿。

取决于Phase 1-2的反馈。如果手动版已经解决问题，可能不需要做到这一步。

关键判断：Phase 1是最重要的。如果手动跑一遍发现江贝说"这个有用"，后面才值得投入开发。如果江贝说"我还是习惯微信聊"，那产品形态就要调整。不要在验证需求之前写代码。

五个核心结论

1. 编辑判断准确——角度全部命中
  • 框架稿主推的三个角度：CDM官改文化、冠军基因下放、用户共创
  • 确认版全部采用了这三个角度，只是表述方式不同
  • 说明编辑大脑的"传播研判"模块逻辑是对的
2. 信息密度差距明显——全部来自上游后补
  • 确认版比框架稿多出的内容：车身三围、轴距、轴荷比、重心高度、底盘减重比例、轮辋名称、屏幕尺寸、座椅材质名称、获奖信息
  • 这些数据全部不在47页活动方案中，是产品部后来单独提供的
  • 编辑大脑正确地标注了这些为"缺口"，但方案本身不包含这些信息
3. 稿件类型需要自动判断
  • 框架稿把活动内容混在产品稿里，确认版是纯产品稿
  • 编辑大脑应增加：同一批素材自动判断"需要出几篇稿、分别什么类型"
  • 本例：应自动拆为"产品上市稿 + 活动新闻稿"两篇
4. 缺口标注完全正确
  • 价格、纳米01Cross信息——标注为硬性缺口 ✅
  • 框架稿正确跳过了信息不足的内容，没有编造
  • "出稿条件判断"机制验证通过：信息不齐→出框架稿而非终稿
5. 写作质量本身没有差距
  • 在信息量相同的段落（如驾控段、智能段），框架稿与确认版文字质量持平
  • 确认版更好是因为信息更多、更精确，不是因为文字更好
  • 如果把全部产品参数喂给brief，输出与确认版差距可缩至10%以内

Phase 1 验证结论

核心假设成立：卡点在信息供给，不在写作能力
编辑大脑的价值 = 把"等信息"从被动等待变成主动可视化管理

待补充验证：以上是内部测试结论。下一步需要拿信息卡+缺口清单+角度建议+框架稿给周杨/江贝看，回答一个问题："如果活动前一周你就收到这些，对你有没有用？"——客户侧的反馈才是真正的Phase 1结论。

八、为什么值得做

对奕派项目
  • 省人：信息提取+brief生成+初稿不再需要专职文案全程参与
  • 提速：从"活动前2-3天赶稿"变成"方案出来当天就有框架稿"
  • 减少废稿：信息齐了才正式出稿，不再写注定被推翻的V1-V3
对其他客户的复制价值
  • 信息提取引擎是通用的——换个客户只需要换知识库和出稿checklist
  • 缺口分析逻辑是通用的——任何公关稿都有"必要信息清单"
  • 编辑研判能力随客户积累越来越强——用得越多越懂这个行业
  • 这就是从"卖人头"到"卖系统"的转型载体

编辑大脑的本质

不是一个写稿工具，是一个"让信息自己找到正确位置"的系统
Skill解决了"怎么写"，编辑大脑解决"写什么、什么时候能写"
